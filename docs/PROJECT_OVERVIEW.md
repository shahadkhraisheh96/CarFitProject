# CarFit — Project Overview

> Read-only exploration produced on the `feature/abd` branch. Nothing in the codebase was modified to produce this document.

CarFit is an **ASP.NET Core 10.0 MVC + Razor Pages** web application for a Jordan-based used-car marketplace. It matches buyers to vehicles through a lifestyle questionnaire, lets sellers list inventory, exposes pre-purchase inspection booking, and provides an admin console.

- **Solution:** `CarFitProject/CarFitProject.slnx` → references `CarFitProject/CarFitProject/CarFitProject.csproj`
- **Target framework:** `net10.0` (`CarFitProject/CarFitProject.csproj:4`)
- **Database:** Microsoft SQL Server (`Server=shahadkh96;Database=CarFit;...`)
- **Auth:** ASP.NET Core Identity (Razor Pages UI), `IdentityUser` + `IdentityRole`
- **Areas:** `Admin`, `Buyer`, `Seller`, `Identity`

### 2.3 Connection strings — where they come from

Two configurations point at the **same** SQL Server:

1. `appsettings.json:3` →
   `Server=shahadkh96;Database=CarFit;Trusted_Connection=True;MultipleActiveResultSets=true;TrustServerCertificate=True;`
   Used by `Program.cs:9` to bind both `ApplicationDbContext` and `CarFitDbContext` via DI. Lines `Program.cs:14-18` defensively append `TrustServerCertificate=True;` if absent.
2. `Models/CarFitDbContext.cs:42` — hardcoded
   `Server=shahadkh96;Database=CarFit;Trusted_Connection=True;TrustServerCertificate=True;`
   inside `OnConfiguring`. EF Core only invokes `OnConfiguring` when options have not been configured, but since both contexts ARE configured by DI in `Program.cs:21-26`, this hardcoded fallback only triggers if the context is constructed parameterless (e.g. by EF tooling). **Still risky** — see §6.

There is no `appsettings.Production.json`. The SQL Server name `shahadkh96` is a developer machine name.

### 2.5 `vw_AvailableCarDetails` & `GetCarMatchesForUser`

Neither object's SQL definition lives in this repository:

- **View** is mapped via `entity.HasNoKey().ToView("vw_AvailableCarDetails")` (`Models/CarFitDbContext.cs:346`). The expected result shape is the union of `Car` + `CarListing` + `Seller` + `InspectionReport` columns plus a `trust_score` and `seller_name` projection — see `VwAvailableCarDetail.cs`.
- **Stored procedure `GetCarMatchesForUser`** is invoked two ways:
  - `Services/RecommendationService.cs:25` — `EXEC GetCarMatchesForUser {0}` (positional parameter named with no `@` prefix, treated as the first positional `@user_id`/`@profile_id` arg by the proc).
  - `Areas/Buyer/Controllers/DashboardController.cs:54-86` — `CommandType.StoredProcedure` with a single `@ProfileId` parameter, reading columns `car_id, make, model, year, listing_price, city, body_condition, description_score, trust_score, DynamicMatchScore`.

  So the actual signature appears to be `GetCarMatchesForUser @ProfileId INT` returning the shape that maps to `RecommendedCarViewModel`. The two callers disagree on what is passed (see §5).

Because neither object is defined in the repo, they must be provided externally (the original SQL Server DB at `shahadkh96`).

## 5. Recommendation Flow

There are **three** parallel recommendation paths, and they don't all use the same input:

### Path A — Buyer Dashboard (the canonical user flow)

1. Logged-in Buyer hits `/Buyer/Dashboard`.
2. `DashboardController.Index(int? activeProfileId)` (`Areas/Buyer/Controllers/DashboardController.cs:22`):
   - Reads `UserProfiles.Where(p.UserId == userId && p.IsActive)`.
   - If none → `RedirectToAction("CreateProfile")`.
   - Selects either the requested profile or the first.
3. Buyer fills the questionnaire on `CreateProfile` view → POST `CreateProfile(UserProfile model)` (line 103):
   - `model.UserId = userId` (claim), `IsActive = true`, defaults filled, `SaveChangesAsync`.
   - Redirects to `Index` with `activeProfileId = model.ProfileId`.
4. `Index` opens a raw ADO.NET `DbCommand` (line 52–88):
   - `CommandText = "GetCarMatchesForUser"`, `CommandType.StoredProcedure`.
   - Binds **`@ProfileId = selectedProfile.ProfileId`** (an `int`).
   - Reads `car_id, make, model, year, listing_price, city, body_condition, description_score, trust_score, DynamicMatchScore` into `List<RecommendedCarViewModel>`.
5. Returns view bound to that list.

### Path B — `IRecommendationService` (defined, not registered)

`Services/RecommendationService.cs:21-27`:
```csharp
public Task<List<RecommendedCarViewModel>> GetMatchesForUserAsync(int userId)
    => _context.Database
        .SqlQueryRaw<RecommendedCarViewModel>("EXEC GetCarMatchesForUser {0}", userId)
        .ToListAsync();
```
- Takes `userId` (int), not `profileId`.
- **Not registered in DI** anywhere in `Program.cs`, and **no controller injects it**. It is dead code unless wired up.

### Path C — `AdvisorController.Match` (manual, profile-agnostic)

`Controllers/AdvisorController.cs:22-99`:
- Anonymous endpoint (no `[Authorize]`).
- `var userProfile = await _context.UserProfiles.FirstOrDefaultAsync(p => p.IsActive == true);` — picks the **first active profile in the whole database**, NOT the current user's. (See §6.)
- Loads `vw_AvailableCarDetails` into memory.
- Scoring (per car, in-memory):
  - Budget within `[BudgetMin, BudgetMax]` → +40 (`totalCriteria += 40`).
  - Transmission matches `TransmissionPref` (case-insensitive) → +30 (`totalCriteria += 30`).
  - If `HasKids && KidsCount > 2`: only adds 30 when `Seats >= 7` or `BodyType.Contains("SUV")`; otherwise it grants the full 30 unconditionally. (`totalCriteria += 30` either way.)
  - Final percentage = `round(matchScore / totalCriteria * 100)`.
  - Keep cars with score ≥ 50, sort desc, render.

## 6. Known Issues

### Confirmed from prior briefing

1. **Hardcoded connection string shadowing DI** — `Models/CarFitDbContext.cs:42`. The DI registration in `Program.cs:25-26` uses `appsettings.json`, but anyone constructing `CarFitDbContext` parameterless (EF tooling, migrations from a different env, accidental `new CarFitDbContext()` in code) will silently bind to `shahadkh96`. Best practice: delete `OnConfiguring` (or guard it with `if (!optionsBuilder.IsConfigured)`), and keep the connection string only in configuration / user secrets.

2. **Admin credentials in source** — `Program.cs:63-64` (`admin@carfit.com` / `AdminPassword123!`). Plus this seeding runs on every startup, so even if an admin changes the password, restarting the app doesn't recreate it (the check is `FindByEmailAsync == null`). Still: a public repo plus this seed = automatic privileged account in any deployment.

3. **CSV BulkLoader naive `line.Split(',')`** — `Areas/Admin/Controllers/AdminController.cs:112`. Any cell containing a comma (escaped or quoted) misaligns columns. No quoting/escaping logic, no `TextFieldParser` or CsvHelper. Also: header row is detected by `lineIndex == 1` only, no header content validation; encoding is whatever `StreamReader` defaults to (UTF-8 BOM only).

4. **`SellerController.Index` SellerId(int) vs userId(GUID) mismatch** — `Areas/Seller/Controllers/SellerController.cs:26`:
   ```csharp
   .Where(l => l.SellerId.ToString() == userId)
   ```
   `SellerId` is `int?`; `userId` is `IdentityUser.Id` (a GUID string like `"3ad1…"`). These never match → seller always sees zero listings. The real bridge between Identity users and the `Sellers` table is missing entirely (no `IdentityUserId` column on `Seller`, no mapping table). Also this controller is in `Areas/Seller/Controllers/` but is missing `[Area("Seller")]`, so its route is `/Seller/...` only because of class-name convention — it's not actually mounted in the Seller area route.

5. **`AdvisorController.Match` uses first global profile** — `Controllers/AdvisorController.cs:25-26` queries `UserProfiles.FirstOrDefaultAsync(p => p.IsActive)` with no user filter and no `[Authorize]`. Whoever visits the page sees recommendations computed from whatever profile happens to be at the top of the table.

### Additional issues found

6. **`Areas/Seller/Inventory/Index` lists every seller's listings** — `Areas/Seller/Controllers/InventoryController.cs:21-29` returns `_context.CarListings.Include(Car).ToListAsync()` with no `Where(SellerId == currentSeller)`. Any logged-in Seller sees the entire marketplace.

7. **`Areas/Seller/Inventory/AddCar` POST never sets `SellerId`** — `InventoryController.cs:50-57` builds a `CarListing` with `CarId`, price, payment method, availability, installment — but `SellerId` stays `null`. Combined with issue 4 and 6, sellers have no real ownership of their listings.

8. **`IRecommendationService` is registered nowhere** — `Services/RecommendationService.cs` defines the interface and impl, but `Program.cs` never calls `AddScoped<IRecommendationService, RecommendationService>()` and nothing injects `IRecommendationService`. Either wire it up or delete it.

9. **Plaintext-named legacy `User.Password`** — `Models/User.cs:14`. The legacy `Users` table (separate from Identity) carries a `Password` column. Even if unused at runtime, its presence is a footgun — and `RecommendationLog.UserId` (int) references it, so any future use of the recommendation log requires the legacy user.

10. **Two parallel user concepts** — `Users` (int PK, separate table) and `AspNetUsers` (GUID, Identity). `UserProfile.UserId` is a string (Identity), but `RecommendationLog.UserId` and `SavedResult.UserId` are int (legacy). Any feature that needs to log a recommendation per logged-in user will need a mapping.

11. **Sync-over-async in startup** — `Program.cs:56,58,67,78,83` (`.Result` / `.Wait()` on `Task`). Works in startup single-thread, but bad pattern; replace with `await` in an async local function.

12. **`TrustServerCertificate=True` everywhere** — defensible for dev but should not survive into prod.

13. **No anti-forgery on `Inventory.GetTermExplanation`** — fine because it's a `GET` returning JSON, but worth noting it's anonymous and unauthenticated.

14. **Empty migration `InitialIdentityTables`** — `Data/Migrations/20260528121455_InitialIdentityTables.cs` has no operations. It exists only to mark the snapshot. Harmless but confusing — likely created by running `dotnet ef migrations add` after schema was already in the DB.

15. **`vw_AvailableCarDetails` and `GetCarMatchesForUser` are not in source control** — they are required at runtime but not defined by any migration. Anyone bootstrapping a fresh DB will not have them, and the app will silently fail on Buyer dashboard / Advisor pages.

16. **`Program.cs` registers `IdentityUser`, not a custom user** — fine, but means there's no way to extend the auth user with name/profile fields without also moving away from `AddDefaultIdentity`.

17. **`InspectionController` injects `CarFitDbContext` in `Index()` but never uses it** — minor; harmless.

18. **`Cars` table has no FK from `CarListings.SellerId` in the model** — the migration defines the FK to `Sellers`, but `CarListing` model + `OnModelCreating` set it up too — OK. Mentioned only because Cars/Sellers/Listings ownership is fragile already.

## 7. Bootstrapping checklist (for someone new)

To run this app from a clean clone, you need:

1. SQL Server reachable via `Server=shahadkh96;...` (or override `DefaultConnection` via user secrets / env var).
2. Apply Identity migrations: `dotnet ef database update --context ApplicationDbContext`.
3. Apply CarFit migrations: `dotnet ef database update --context CarFitDbContext`.
4. Create the **missing** SQL view `vw_AvailableCarDetails` and stored procedure `GetCarMatchesForUser @ProfileId INT` against the CarFit DB — neither lives in source.
5. Run the project (`dotnet run`) — the admin user `admin@carfit.com` / `AdminPassword123!` will be seeded automatically.
